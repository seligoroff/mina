# Инструкции для генерации протокола из выгрузки Telegram (JSONL)

Цель: по выгрузке переписки (формат JSONL: по сообщению на строку) сформировать структурированный протокол встречи/обсуждения.

Входные данные (каждая строка — JSON-объект):
- id: целое
- date: ISO8601 с таймзоной, например 2025-11-14T06:25:51+00:00
- from_id: целое или null
- from_name: строка с именем отправителя или null
- text: текст сообщения (может содержать переносы строк)
- reply_to: id сообщения, на которое есть ответ, или null
- entities: массив объектов (ссылки, упоминания) — можно игнорировать при анализе содержания
- attachments: массив объектов вложений — не использовать, если не требуется

Правила:
1) Игнорировать служебные сообщения и пустой/шумный контент (реакции без смысла, стикеры без текста, системные события).
2) Ссылки приводить только по необходимости (например, если в сообщении есть ключевая ссылка на документ).
3) Стараться группировать сообщения по темам и логике обсуждения. Учитывать `reply_to`, чтобы понимать нити обсуждения.
4) Время сообщений использовать для понимания порядка и временных рамок. Если период охватывает несколько дней, важно отражать динамику по дням (что происходило в какой день).
5) Обрабатывать язык: текст может быть на русском; сохранять смысл оригинала, избегать вольных интерпретаций.
6) Если встреча неочевидна, всё равно формировать протокол: темы, решения, действия.

Формат результата (Markdown, на русском):

## Протокол обсуждения
- Период: <YYYY-MM-DD HH:MM> — <YYYY-MM-DD HH:MM> (UTC)
- Канал/чат: Telegram (имя/идентификатор, если известно)
- Участники (по именам из from_name, если доступны): Имена через запятую.
  - Важно: не пытайся «склеивать» или нормализовывать имена разных участников. Считай разными людьми, например, `igor` и `Игорь`, даже если они звучат похоже. Используй имя из поля `from_name` без объединения по смыслу.
- Если период охватывает несколько дней — в тексте протокола явно разделяй блоки по дням (например, «День 1 — 2025-11-14», «День 2 — 2025-11-15») и группируй события внутри дня по темам.

### Повестка / Темы
- Краткий список ключевых тем обсуждения (по 1–2 строки на тему).

### Краткое резюме
- 3–7 буллетов, отражающих главное из обсуждения.

### Подробности по темам и дням
Для многодневной переписки рекомендуется структура:

#### День N — YYYY-MM-DD
1. Тема A
   - Суть: 1–3 предложения
   - Аргументы/замечания: 2–5 буллетов
   - Итоги/решения: 1–3 буллета
2. Тема B
   - ...

Если переписка за один день, можно опустить уровень «День N» и оставить только тематические блоки.

### Решения
- Чёткий список принятых решений (1–7 пунктов). Формулировать утвердительно.

### Action items (задачи)
- [ ] Краткое описание задачи — Ответственный (Имя) — Дедлайн (если указан) — Ссылка (если релевантно)
- [ ] ...

### Риски / Открытые вопросы
- Перечислить выявленные риски или вопросы, требующие отдельного прояснения.

### Приложения (опционально)
- Ссылки на документы, артефакты, чаты (если они критичны).

Примечания:
- Не писать дословные цитаты, если они не несут уникальной ценности.
- Избегать лишней технической детализации, если она не влияет на решения/результат.
- Сохранять нейтральный, деловой стиль.

Если в выгрузке за указанный период нет содержательного обсуждения, кратко зафиксировать это: “Существенных сообщений за период не обнаружено”.

### Технические вопросы, вызвавшие споры

- Перечислить технические вопросы, вызвавшие споры в обсуждении. 
- Указать, чью сторону в споре ты выбрал бы, дать обоснование.

**Пример:**

```
### 1. Документирование ошибок в OpenAPI
**Суть спора:** Igor selivanov утверждал, что команда не выполняет "джентльменское соглашение" по актуализации ошибок в OpenAPI. Александр Андрухов возражал, что проблема в постановке задач и их восприятии командой.

**Чью сторону выбрал бы и почему:** Сторону igor selivanov. Проблема документирования API является критически важной для проекта, и ожидание того, что команда будет самостоятельно следить за актуализацией ошибок во всех эндпоинтах, оправдано для профессиональной разработки.
```
